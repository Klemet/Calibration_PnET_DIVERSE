Initial species parameters

Here, I describe how/from where I obtained these initial parameter values that serve as the basis of the calibration. These parameter values are the first ones were are going to use, before tweaking them so that we can match empirical grow data or expected outcomes of competition, climate change, etc. in the follwing sections.

Tip

If you cannot find good empirical values or studies related to the parameters of your species, the calibration tips in Gustafson and Miranda (2023) give a lot of suggestion for initial parameter values.

Tip

If you have enough measurements to use for some of the more precise and ecophisiological parameters, you might want to use measurements that are close to your study landscape. See Some important notions before we start for more.

What parameters are we talking about here, and what are the different categories of parameters ?¶

In this page, we will talk about :

  • Core species parameters : The species parameters used by the LANDIS-II core (e.g. Longevity, dispersal distance, etc.). These will go in the “core” species parameter file for LANDIS-II (indicated in the main scenario file).

  • PnET species parameters : The species parameters used by PnET Succession that will vary between species. These will go into the species parameters file of PnET Succession.

  • PnET generic species parameters : The species parameters used by PnET Succession that we will keep constant between species. These will go into the generic parameter file of PnET Succession. See below for recommendations on which parameters to put in this category.

  • PnET generic parameters : Other generic parameters of PnET Succession that are not related to species, but that are still crucial for the functionning of PnET Succession (PARunits, PrecipEvents, etc.).

Why should some species parameters be “generic” and constant accross species in PnET Succession ?

Gustafson and Miranda (2023) advises that several species parameters should be used as generic species parameters, meaning that their value should be the same between species, in order to facilitate the comprehension of competition processes. Quoting from them :

Use the GenericPnETSpeciesParameters file for all parameters that are not varied among species. Set parameters that will be held constant for your particular experiment or study (e.g., MaintResp, InitialNSC, etc.)

Parameters that the model is highly sensitive to (for which common values should be used as much as possible) include: MaintResp, AmaxA, AmaxB, k, and FracBelowG.

Minimizing species differences in parameters such as SLWDel, PsnAgeRed, k, MaintResp, DNSC and FracBelowG, will make competitive interactions more predictable. Hold these as close to each other as possible, varying other parameters to calibrate as much as possible within empirical limits.

Which PnET species parameters should be generic ?

The choice depends on your species. As recommanded above by Gustafson and Miranda (2023), using as many parameters as you can as constant and generic accross species will make competition and other processes in PnET Succession easier to interpret and understand. But sometimes, you will have to change these parameters between species so as to properly represent their differences.

Here, I will start with the recommandations of Gustafson and Miranda (2023) and make MaintResp, InitialNSC, k, FracBelowG, PsnAgeRed, DNSC similar between species. SLWDel, k, AmaxA and AmaxB will not be generic as they have to vary at least between evergreen and deciduous according to Gustafson and Miranda (2023).

The rest of the species parameters (see here for a full list) like FolN, SLWmax, TOFol, TORoot, TOWood, HalfSat, H1, H2, H3, H4, LeafOnMinT, PsnTMin, PsnTOpt, EstMoist, EstRad, FracFol, FrActWd, SLWDel, k, AmaxA and AmaxB will change by species or by growth form/strategy (e.g. shade tolerant, etc.). Optional parameters (such as PsnTMax, ColTol, MaxLAI, etc.) will not be considered here.

Format of the following tables of parameters

As the end of this notebook is a code section that will read the following tables of parameters and transform them in Python dictionnaries that can be used by the functions of functionsForCalibration.py to create LANDIS-II parameter files, which will be used to automatize LANDIS-II simulations further on.

So that the function can read the tables properly, the content of each cell must respect the following format :

  • The cell must start with the value of the parameter

  • All other content must be either in footnote format (between square braquets []), between parenthesis, or after an asterisk. This content will then be ignored by the function to just keep the value at the beginning of the cell.

Example :

  • 0.35 (average among species in [^2] and [^3]) : Good. Only the value 0.35 will be taken; the rest, between parenthesis, will be ignored.

  • 0.35 : average among species in [^2] and [^3] : Not good. The rest of the cell appart from the value (0.35) should be between parenthesis or square braquets.

About the LayerThreshRatio parameter

LayerThreshRatio is a generic parameter that is used to define how easily age cohorts of the same species are groupped in the same canopy layer.

I’m still having trouble wrapping my mind about how the parameter works exactly; what’s certain is that it has a large effect on the outcomes of simulations. Increasing it seems to let more cohorts cohabitate together in a cell, especially young and old cohorts. Might be because in this case, young cohorts tend to be grouped with older cohorts more easily, and they thus receive more light according to the algorithm.

It’s difficult for me to recommand the right value for this parameter. For now, I will put it at 0.67, as it’s one of the values recommanded by Eric Gustafson in the user guide, and because it seems to allow more room for complex dynamics inside the cells rather than a periodic domination by large, old cohorts. But you might want to change it if you’re seeing strange results of cohabitations between cohort in a given cell.

Exporting initial parameters into JSON files¶

These .JSON files will simply contain the values of the tables above, and will be use to quickly launch LANDIS-II simulations with these parameters without doing it by hand (see next notebooks).

from functionsForCalibration import *
import json

# We read the markdown content of the previous cell that contains the markdown table with the parameters,
# and we transform it into a Python dictionnary that can be used with the other functions to interact with
# LANDIS-II scenarios.

# We add the Landis Data parameter indication
coreSpeciesParameters = dict()
coreSpeciesParameters['LandisData'] = 'Species'
markdownText = read_markdown_cell("./3.Initial_Species_Parameters.ipynb", 10)
markdownTable = extract_table(markdownText)
coreSpeciesParameters.update(parseTableSpeciesParameters(markdownTable))

# print(coreSpeciesParameters)

# Reading PnET Species Parameters
# We format the dictionnary for use with the other functions
PnETSpeciesParametersFinal = dict()
PnETSpeciesParametersFinal ['LandisData'] = 'PnETSpeciesParameters'
markdownText = read_markdown_cell("./3.Initial_Species_Parameters.ipynb", 14)
markdownTable = extract_table(markdownText)
PnETSpeciesParameters = parseTableSpeciesParameters(markdownTable)
PnETSpeciesParametersFinal['PnETSpeciesParameters'] = PnETSpeciesParameters

# print(PnETSpeciesParametersFinal)

# Species generic parameters
# We add the Landis Data parameter indicator and another needed for this parameter file
genericParameters = dict()
genericParameters['LandisData'] = 'PnETGenericParameters'
genericParameters['PnETGenericParameters'] = 'Value'
markdownText = read_markdown_cell("./3.Initial_Species_Parameters.ipynb", 12)
markdownTable = extract_table(markdownText)
genericParameters.update(parseTableGenericParameters(markdownTable))

# print(genericParameters)

# Generic parameters
# We're going to add these to the generic species parameters to be in one single dict to create the parameter file that corresponds to them
markdownText = read_markdown_cell("./3.Initial_Species_Parameters.ipynb", 16)
markdownTable = extract_table(markdownText)
genericParameters.update(parseTableGenericParameters(markdownTable))
# print(genericParameters)

# Now, we change any mention of the species name toward a species code usable in LANDIS-II
# We change this notation to use the species code system from the National Forest Inventory of Canada,
# as this nomenclature is used at other places in these notebooks
# See https://nfi.nfis.org/resources/general/3-TreeSpeciesList-Version4.5.pdf
# Since we're going to re-use this equivalence several time, I'm using a file in which
# the equivalence is writen
with open("./ReferencesAndData/NFI_Data/NFI_SpeciesNameEquivalence.json", 'r') as file:
    speciesCodeDict = json.load(file) 

for key in speciesCodeDict.keys():
    coreSpeciesParameters = replace_in_dict(coreSpeciesParameters, key, speciesCodeDict[key])
    PnETSpeciesParametersFinal = replace_in_dict(PnETSpeciesParametersFinal, key, speciesCodeDict[key])

# We print the dictionnaries for verification
print(coreSpeciesParameters)
print(PnETSpeciesParametersFinal)
print(genericParameters)

# We save the dictionnary in a file to use it later in other notebooks.
with open('./SpeciesParametersSets/Initial/initialCoreSpeciesParameters.json', 'w') as json_file:
    # Step 4: Write the dictionary to the file
    json.dump(coreSpeciesParameters, json_file)
with open('./SpeciesParametersSets/Initial/initialPnETSpeciesParameters.json', 'w') as json_file:
    # Step 4: Write the dictionary to the file
    json.dump(PnETSpeciesParametersFinal, json_file)
with open('./SpeciesParametersSets/Initial/InitialGenericParameters.json', 'w') as json_file:
    # Step 4: Write the dictionary to the file
    json.dump(genericParameters, json_file)
